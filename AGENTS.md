# AGENTS.md

## 🎯 角色定位

你是一个资深前端架构师，负责维护一个微信小程序项目。
你必须始终遵守以下规则进行分析、修改和生成代码。

你的目标不是“尽快写完代码”，而是：

- 保持目录清晰
- 保持组件拆分合理
- 保持样式体系统一
- 保持代码最小改动
- 优先复用已有能力，禁止重复造轮子

---

## 🧠 修改代码前必须先执行的步骤

在修改任何代码前，必须先完成以下检查：

1. 先分析当前页面所在目录结构
2. 先识别当前页面已有的子组件
3. 先确认是否已有可复用组件或公共样式变量
4. 先判断修改应该发生在：
   - 页面层
   - 子组件层
   - 公共样式层
   - 全局配置层

未完成以上分析前，不允许直接输出修改代码。

---

## 📁 项目目录理解规则

这是一个微信小程序项目，必须遵守小程序目录习惯，不得擅自套用 Vue / React 项目结构。

### 页面目录规则

每个页面放在 `pages/<page-name>/` 下，例如：

- `pages/home/home.js`
- `pages/home/home.json`
- `pages/home/home.scss`
- `pages/home/home.wxml`

页面目录中的这 4 个文件分别承担固定职责：

- `*.js`：页面逻辑、数据、事件
- `*.json`：页面配置
- `*.scss`：页面样式
- `*.wxml`：页面结构

禁止把页面写成 `index.*` 风格，除非项目中已经明确采用该约定。

---

## 🧩 页面与组件职责划分

### 页面文件职责

页面主文件只负责：

- 页面整体布局
- 页面级数据组织
- 子组件组合
- 子组件通信
- 页面生命周期

页面层不应堆积过多重复 UI 细节。

### 子组件职责

`pages/<page-name>/components/` 下的组件用于承载可拆分的独立模块，例如：

- banner
- card
- row
- grid

当某个区块满足以下任一条件时，应拆成组件：

1. 结构相对独立
2. 可能复用
3. 模板超过一个清晰功能块
4. 有独立样式和交互

---

## 📦 当前项目组件拆分偏好

参考当前目录，例如 `pages/home/components/` 下已有：

- `order-card`
- `promo-row`
- `shortcut-grid`
- `top-banner`
- `vehicle-card`

因此后续修改 `home` 页时，必须优先检查是否应在这些组件内修改，而不是直接把所有内容堆回 `home.wxml`。

规则如下：

- 能在现有组件内改，就不要回写到页面层
- 能复用已有卡片结构，就不要新建相似组件
- 新增区块前，先判断是否与现有 `card / row / banner / grid` 结构一致

## 🎨 样式系统规则

### 样式变量使用规则

项目已有公共样式变量目录：

- `styles/variables/_border.scss`
- `styles/variables/_color.scss`
- `styles/variables/_layout.scss`
- `styles/variables/_motion.scss`
- `styles/variables/_padding.scss`
- `styles/variables/_size.scss`
- `styles/variables/_typography.scss`

因此所有样式开发必须遵守：

1. 颜色、字号、圆角、间距、动效等，优先使用已有变量
2. 禁止随意新增临时魔法值
3. 禁止同一类视觉属性在多个地方重复硬编码
4. 新增样式前，先查已有变量体系是否可复用

### 引入规则

样式变量必须通过引入方式使用，不允许直接复制变量值到页面样式中。

---

## 📐 布局规则

编写页面和组件样式时，必须优先使用：

- flex
- grid

目标：

- 保证结构稳定
- 保证对齐关系清晰
- 保证适配性更好

禁止行为：

- 禁止为了对齐大量使用绝对定位
- 禁止依赖大量写死宽高、margin、padding 来“凑布局”
- 禁止用死数据模拟响应式效果
- 禁止为了局部视觉效果破坏整体布局体系

如果确实需要固定尺寸，必须先判断该尺寸是否属于：

- 设计规范要求
- 图片资源固有比例
- 小程序控件限制

---

## ✍️ 代码风格规则

### JavaScript

- 保持逻辑清晰，命名语义化
- 页面层只保留页面级逻辑
- 重复逻辑要考虑抽离
- 事件函数命名必须体现行为目的
- 不允许无意义缩写

### WXML

- 结构分层清晰
- 不要把过长模板堆在一个文件里
- 有明显功能边界的区块优先组件化
- class 命名应体现模块语义

### SCSS

- 样式按模块组织
- 避免层级过深
- 避免无意义嵌套
- 禁止把页面样式和组件样式互相污染

---

## 🚫 严格禁止的行为

- 禁止无分析直接改代码
- 禁止为了省事把所有代码堆到页面主文件
- 禁止新增重复组件
- 禁止绕过公共变量直接写颜色、字号、圆角、间距
- 禁止大范围重构与当前需求无关的代码
- 禁止擅自修改项目目录约定
- 禁止把 Vue / React 的开发习惯直接套到小程序项目中
- 禁止只追求“能跑”，忽略结构和维护性

## 注意事项

- 对于存在border的元素，要使用怪异盒子模型，避免切换时元素抖动
